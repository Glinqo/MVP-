# AGENTS.md

## Git 自动同步准则

- 以后每次完成“有效改进”后，必须把改动同步到 Git。有效改进包括：新增功能、修复缺陷、补充知识库/题库/资源、优化提示词、调整 API/前端体验、更新关键文档或测试。
- 同步前必须先运行与改动类型匹配的验证：JSON 解析、JS 语法/测试、Python 编译/API 冒烟、评分测试或必要的手工检查。
- 只提交本次任务相关文件；如果工作区里有用户或运行时产生的无关变更，必须保留但不要混入提交。
- 提交信息要简短说明实际改进，例如 `Improve student graph evidence view`、`Expand PLC troubleshooting questions`。
- 若远端已配置，验证通过后自动执行 `git push` 同步到远端；若没有远端、认证失败或推送被拒绝，需要在最终回复中明确说明阻塞原因和本地提交号。
- 禁止使用 `git reset --hard`、强制推送或覆盖远端历史，除非用户明确要求。
- 不提交密钥、Token、临时缓存、运行日志、个人隐私或大型无关文件。

本仓库是轻量 MVP，不追求完整平台化。当前以自研本地 Web/API Demo 为主，星辰 Agent 相关文件仅作为可选兼容和历史资产保留。

## 工作原则

- 优先修改 Markdown、JSON 和少量评分代码。
- 可以新增轻量后端、前端或 CLI，但必须服务当前实训闭环，不做大系统。
- 优先使用 JSON/Markdown 作为主数据源；如需持久化，先用本地文件，确有必要再考虑 SQLite。
- 保持文件可复制、可阅读、可在本地 Demo 和可选平台工作流中复用。
- JSON 必须可被标准解析器读取，不使用注释和尾逗号。
- 评分逻辑要稳定、可解释，优先规则表，不依赖黑盒模型。

## 推荐改动顺序

1. 修改 `docs/product_scope.md` 明确边界。
2. 修改 `knowledge/ability_nodes.json` 和 `knowledge_50.*`。
3. 修改 `diagnosis/*.json`。
4. 同步更新评分代码模块和本地调用入口。
5. 运行 `node tests/scoring.test.js`。

## 不做

- 不搭建登录、权限、班级管理、数据库。
- 不实现大型知识图谱平台。
- 不把所有教学内容一次性塞进提示词。
- 不把评分规则写死在提示词里。
