# 2026-07-11 开发总结

## 概述

完成模块一（对话与产品流程）的全部功能开发及多轮迭代修复。从"所有输入走同一条规则匹配路径"升级为"意图分类 → 路由分发 → LLM 增强 → 对话内卡片渲染 → 会话持久化"的完整智能体对话管道。共新增 3 个 Python 模块、修改 5 个现有文件、产出 3 份文档。

## 一、四大开发阶段

### Phase 1：意图分类管道

**新增** `app/services/intent.py`：
- LLM 优先 + 加权关键词兜底的双路意图分类器
- 6 种意图：diagnosis / knowledge_qa / quiz / graph / learning_path / clarify
- 强信号权重 2（如"帮我出"→quiz），弱信号权重 0.3（如"PLC"→diagnosis）
- diagnosis 平局时主动让位避免技术通用词误触发
- 无任何关键词匹配时默认归类为 clarify

**新增** `app/services/intent_handlers.py`：
- 5 个意图处理器，均返回与原有 `chat_message()` 一致的响应结构
- `handle_knowledge_qa`：RAG 检索 → LLM 基于检索生成带来源的回答
- `handle_quiz`：调用 `personalized_quiz()` 生成个性化练习题
- `handle_graph`：调用 `build_ability_graph()` 列出能力节点
- `handle_learning_path`：调用 `personalized_plan()` 生成训练单和方案
- `handle_clarify`：LLM 生成口语化追问，3 轮限流，超限后最佳诊断兜底

**修改** `app/services/chat.py`：
- `chat_message()` 入口新增意图分类步骤 → 按 intent 分流到处理器
- diagnosis/clarify 继续走原有 assist 流程，零改动零回归
- 新增 `_finalize()` 为非 diagnosis 意图统一记录 session 事件

### Phase 2：LLM 驱动的澄清追问

- `generate_clarify_questions()` 加载 `prompts/clarify_prompt.md` 生成自然追问
- 双路径覆盖：`clarify` 意图独立路由 + diagnosis 路径 `need_clarification` LLM 增强
- 3 轮限流机制：`_count_clarify_rounds()` 基于 session events 计轮
- `_best_effort_from_clarify()` 超限后给出有依据的参考判断
- LLM 不可用时自动回退到 JSON 静态追问

### Phase 3：对话内卡片组件

**修改** `web/app.js`：
- `renderMessageExtras()` 重构为 `renderMessageCards()`，拆分为 5 种 inline 组件：
  - `renderEvidenceStrip()`：圆角色标标签条（橙/蓝/绿/紫颜色编码）
  - `renderReasoningBar()`：编号步骤 → 箭头连接的推理链
  - `renderKnowledgeCards()`：蓝色可展开卡片，带来源引用和"追问"按钮
  - `renderAbilityCards()`：黄色卡片，能力名+命中标签+"问 AI 讲解"按钮
  - `renderTaskCards()`：绿色卡片，任务类型+时长+"开始任务"按钮
- 卡片按钮通过 `attachAskButtons()` 绑定到 Explain Drawer
- 知识卡片 `explainType` 优先级链修正：question_id → ability_id → knowledge_id → message

**修改** `web/styles.css`：
- 新增约 150 行卡片样式（`.chat-cards` / `.kb-card-inline` / `.ability-card-inline` / `.task-card-inline` / `.evidence-strip` / `.reasoning-bar`）

### Phase 4：协议文档与状态建模

**新增** `docs/模块开发日志/功能调用协议.md`：
- 8 个后端模块的完整接口签名（输入/输出 JSON Schema）

**新增** `docs/模块开发日志/对话状态结构.md`：

**迁移** `docs/模块一_对话与产品流程_缺失功能分析.md` → `docs/模块开发日志/模块一_缺失功能分析.md`：

## 二、关键问题修复

### 修复 1：Explain Drawer 知识/能力讲解从静态模板改为 LLM 动态生成

**问题**：点击知识卡片或能力卡片的"追问"/"问 AI 讲解"按钮，Explain Drawer 中的讲解文本来自硬编码 JSON 字段，每次内容完全相同。

**修复** `app/services/explanation.py`：
- 新增 `_llm_explain()` 辅助函数和 `_build_learner_snapshot()` 学生状态提取
- `knowledge_explanation()`：LLM 收到知识内容 + 常见错误 + 学生薄弱点 → 生成个性化讲解
- `ability_explanation()`：LLM 收到能力描述 + 前置依赖 + 学生状态 → 生成师傅带教式讲解
- `message_explanation()`：LLM 收到匹配现象 + 排查步骤 + 候选故障 → 生成场景化讲解
- 三个 handler 的 `reasoning_steps` 改为面向学生的行动指南（而非 AI 内部推理过程）
- 三个 handler 的 `suggested_questions` 改为引用具体知识/能力名称的上下文问题
- LLM 不可用时自动回退原模板

### 修复 2：澄清追问传入对话历史，追问不再重复

**问题**：第二轮追问与第一轮内容几乎相同，系统不知道学生刚才回答了什么。

**修复** `app/services/intent_handlers.py` + `app/services/chat.py`：
- `generate_clarify_questions()` 新增 `history` 参数
- LLM prompt 中注入最近 4 轮对话的 `recent_conversation` 字段
- 明确指示 LLM："只追问仍然未知的信息，不要重复问已经回答过的"
- `handle_clarify()` 和 `chat.py` 诊断路径均传入对话历史

### 修复 3：知识卡片"追问"按钮误路由到 message_explanation

**问题**：点击知识卡片的"追问"按钮，Explain Drawer 标题始终是"传感器灯亮但 PLC 输入灯不亮"（原诊断标题），而非知识点名称。

**修复** `web/app.js`：
- `attachAskButtons()` 中 `explainType` 优先级链新增 `knowledge_id` 判断
- 之前：question_id? → ability_id? → message
- 之后：question_id? → ability_id? → knowledge_id? → message

### 修复 4：会话持久化，刷新不丢失对话

**问题**：刷新网页后对话历史丢失，需要从空白会话重新开始。

**修复** `web/app.js` + `web/index.html`：
- `sessionId` 从 `localStorage` 读取，刷新后复用同一会话
- `messages` 每次 `addMessage()` 后自动保存到 `localStorage`（最多 40 条）
- `boot()` 启动时从 `localStorage` 恢复消息
- 顶部栏新增"新对话"按钮，清除记录并重新开始
