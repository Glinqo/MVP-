## 二、逐项缺失分析

### 2.1 🔶 多轮对话与上下文记忆

**功能书要求**：
> 实现多轮对话和上下文记忆

**当前实现**：
- `chat.py`：将最近 8 轮 `history` 传给 LLM；`classify_intent()` 利用历史做意图分类
- `feedback.py`：以 session 为单位持久化事件到 JSON 文件
- `learner_context.py`：从 session events 汇总学生能力状态

**缺失**：
- ⬜ **主动上下文注入**：LLM 只收到裸 history，未在 system prompt 中注入"你上次薄弱的是 X""你之前学过 Y"。`learner_context_pack()` 已计算好了薄弱能力，但 `build_user_prompt()` 只把 context 放在 user message 末尾。
- ⬜ **上下文遗忘模拟**：模块四提到"考虑学习时间和遗忘因素"，对话层完全未考虑时间衰减。
- ⬜ **跨 session 记忆**：每次新 session 从零开始，无法引用历史 session 的学习记录。

**建议优先级**：中

### 2.2 ✅ 识别模糊提问并进行必要追问

**功能书要求**：
> 识别模糊提问并进行必要追问；赛题要求 2—3 轮追问，以清晰、口语化的方式引导用户

**当前实现（2026-07-11 完成）**：

两条路径均支持 LLM 驱动的自然追问：

| 路径 | 触发条件 | 实现 |
|---|---|---|
| `clarify` 意图路由 | 输入模糊/无技术关键词（"能帮我看看吗""怎么办"） | `handle_clarify`：LLM 生成口语化追问 + 3 轮后最佳诊断 |
| diagnosis 路径 need_clarification 分支 | 信息不足以诊断 | `chat.py` 调用 `generate_clarify_questions` 用 LLM 替代静态追问 |

**实现细节**：

| 子功能 | 标记 | 文件 | 说明 |
|---|---|---|---|
| LLM 驱动追问 | ✅ | `intent_handlers.py:generate_clarify_questions()` | 加载 `clarify_prompt.md`，基于已知/缺失信息生成自然追问 |
| 追问轮数追踪 | ✅ | `intent_handlers.py:_count_clarify_rounds()` | 基于 session events 计数，最大 3 轮 |
| 最大轮后最佳诊断 | ✅ | `intent_handlers.py:_best_effort_from_clarify()` | 3 轮后给出有依据的参考判断 + 下一步建议 |
| 关键字兜底 | ✅ | `intent_handlers.py` | LLM 不可用时回退到 JSON 静态追问 |
| clarify 独立路由 | ✅ | `chat.py` | `clarify` 意图 → `handle_clarify(payload, intent_result)` |
| diagnosis 路径 LLM 追问 | ✅ | `chat.py:267-290` | `need_clarification` 时调用 `generate_clarify_questions` |
| 口语化追问 | ✅ | LLM 生成 | 追问风格自然，类似实训师傅带教口吻 |
| clarify 意图关键词 | ✅ | `intent.py` | "帮我看看"权重2，"怎么办/不知道/不懂"权重1；无匹配默认 clarify |

**验证结果**：
- ✅ 模糊输入 "能帮我看看吗" → `clarify` intent → 生成 3 个具体追问
- ✅ 部分信息 "传感器灯亮了但PLC没反应" → `diagnosis` intent → need_clarification → LLM 追问
- ✅ 3 轮后自动给出最佳诊断（"我已经问了 3 轮，先根据现有信息给你一个参考判断"）
- ✅ LLM 不可用时自动回退到 JSON 静态追问
- ✅ 冒烟测试 54 断言全部通过

**建议优先级**：~~高~~ → ✅ 已解决

### 2.3 ✅ 识别学生当前需求（意图分类）

**功能书要求**：
> 识别学生当前需求，如知识咨询、故障排查、能力查询、训练请求

**当前实现（2026-07-11 完成）**：

| 文件 | 职责 | 关键设计 |
|---|---|---|
| `app/services/intent.py` | 双路意图分类器 | LLM 优先 + 加权关键词兜底（强信号权重2/弱信号0.3，diagnosis 平局让位） |
| `app/services/intent_handlers.py` | 4 个意图处理器 | knowledge_qa → RAG+LLM；quiz → personalized_quiz；graph → build_ability_graph；learning_path → personalized_plan |
| `app/services/chat.py` | 路由编排 | `chat_message()` → `classify_intent()` → 分流到 handler 或保留原 assist 流程 |

**路由表**：

| 意图 | 示例输入 | 路由目标 | 关键产物 |
|---|---|---|---|
| `diagnosis` ✅ | "传感器灯亮但PLC没反应" | 原 `assist()` 流程 | 故障诊断 + 能力命中 + 知识缺口 + 补救任务 |
| `knowledge_qa` ✅ | "什么是NPN和PNP的区别" | `handle_knowledge_qa` | RAG 检索 + LLM 回答 + 来源引用 |
| `quiz` ✅ | "帮我出几道PLC的题" | `handle_quiz` | 个性化练习题 |
| `graph` ✅ | "需要哪些岗位能力" | `handle_graph` | 岗位能力图谱 + 节点列表 |
| `learning_path` ✅ | "我下一步应该学什么" | `handle_learning_path` | 个人培养方案 + 今日训练单 |
| `clarify` ✅ | "能帮我看看吗" | `handle_clarify` | LLM 自然追问 + 3 轮限流 |
| `teacher_view` ⬜ | — | 未接入 | — |

**验证结果**：
- ✅ 冒烟测试 `api_smoke.test.py`：54 个断言全部通过，diagnosis 路径零回归
- ✅ 意图分类准确率：12/12 典型输入全部分类正确
- ✅ 集成测试：5 种意图均正确路由并返回有效回答
- ✅ Fallback：LLM 不可用时，关键词分类兜底 + 规则回答兜底

**仍缺失**：
- ⬜ `teacher_view` 意图接入

**建议优先级**：~~高~~ → 核心已解决，剩余为中

### 2.8 🔶 LLM 失败时的规则兜底

**功能书要求**：
> 完善大模型调用失败时的规则兜底机制

**当前实现**：
- ✅ `chat.py`：LLM 不可用 → `fallback_answer()`（diagnosis 路径）
- ✅ `intent.py`：LLM 分类失败 → 关键词兜底
- ✅ `intent_handlers.py`：LLM 不可用 → 知识库原文 / 模板文本
- ✅ 前端显示 `fallback_used` 标记

**缺失**：
- ⬜ **LLM 错误分类不足**：只记录 `str(exc)`，不区分 API key 无效/超时/过载/内容过滤
- ⬜ **无三级降级**：只有"LLM → 规则"两级，缺中间层

**建议优先级**：低
