# 学生端增强路线与借鉴方案

## 当前自我批判

现有 MVP 已经具备聊天、问题排故、双能力图谱、讲题、自测、实训任务、个人培养方案和教师摘要，但学生端仍有几个明显短板：

1. 功能入口较多，学生不一定知道此刻应该先看图谱、做自测、看讲解还是做实训。
2. 个人能力图谱已经能更新，但缺少一个面向学生的“行动翻译层”，即把图谱状态转成今天先做什么。
3. 讲题、场景训练、培养方案之间有联动，但还可以更像一位师傅：先判断风险，再安排一个短任务，再用复测确认。
4. 图谱可视化能展示节点，但对“岗位要求 vs 个人差距”的对比还不够直观。
5. 知识库已有来源字段，但资源推荐仍偏静态，后续应根据薄弱能力、学习偏好和当前故障现象排序。

## 已落地增强：学习驾驶舱

本轮新增“学生学习驾驶舱”，目的是把分散功能压缩成一个总控面板。

新增能力：

- `GET /api/student/dashboard?session_id=...`
- 前端新增“学习驾驶舱”全屏工具入口。
- 驾驶舱展示岗位准备度、状态统计、风险提醒、马上处理的能力、今日动作、推荐工具和最近证据。
- 聊天、自测、讲题、反馈和角色扮演后自动刷新驾驶舱。
- 所有判断来自个人能力图谱、学习事件和确定性规则，不让 LLM 自由评价学生能力。

借鉴来源：

- Inno Agent：learner context pack，把学习者状态压缩后注入主流程。
- DeepTutor：mastery path，把薄弱能力变成学习路径。
- EduAdapt AI：progress tracker，把学习事件转成可读进度。
- PersonalizedAdaptiveLearning：weak-topic ordering，把薄弱主题排序后推荐下一步。

## 可继续借鉴的项目能力

### KnowBook AI：图谱节点解释与路径证据

可借鉴点：

- 点击图谱节点时，不只显示节点详情，还显示“它和前后节点的路径关系”。
- 回答时把引用知识、图谱节点和练习任务一起返回。

本项目下一步做法：

- 节点详情加入“上游前置能力”和“下游岗位任务”。
- 当前问题图谱显示“为什么这个问题命中这些能力”的路径解释。

### Educational_RAG_System：父子块和混合检索

可借鉴点：

- 父块保存完整上下文，子块用于精准召回。
- 关键词、向量、规则多路召回后重排。

本项目下一步做法：

- 不引入向量数据库，先把知识库拆成 `symptom`、`wiring`、`monitoring`、`program`、`pneumatic` 等轻量索引字段。
- 检索结果显示命中词、命中能力和来源，方便教师复核。

### DeepTutor：讲题后的掌握路径

可借鉴点：

- 题目讲解不是终点，讲完后继续推进 mastery path。

本项目下一步做法：

- 讲题抽屉底部增加“1 分钟复问”，让学生用一句话解释为什么公共端会影响输入。
- 复问结果不由 LLM 打分，先用关键词/rubric 标记为 `improving` 或 `needs_review`。

### EduAdapt / PAL：自适应学习计划

可借鉴点：

- 根据弱点、进度和资源生成阶段性路径。
- 弱点排序和资源匹配比单纯推荐资源更重要。

本项目下一步做法：

- 培养方案加入“完成条件”和“复测题”，每个阶段明确什么时候可以进入下一阶段。
- 资源推荐加入能力匹配分：同一能力优先视频，其次文本，最后任务说明。

### Multi-Agent Study Assistant：专职角色分工

可借鉴点：

- 分析者、路线规划者、测验生成者、导师、资源查找者各自负责一个任务。

本项目取舍：

- 不引入多 Agent runtime，避免慢和不可控。
- 保持服务级分工：`chat`、`retrieval`、`graph`、`explanation`、`plan`、`quiz`、`scenario`。
- 后续可以把每个服务的 prompt 和规则写得更像专职 Agent，但共享同一个本地状态。

### Asset Intelligence Graph-RAG：工业实体关系

可借鉴点：

- 工业对象不是孤立知识点，而是设备、部件、规格、兼容关系和文档证据的网络。

本项目下一步做法：

- 把当前“能力-知识-任务”图谱扩展为“现象-设备部件-检测证据-可能原因-排故步骤-能力缺口”轻量关系。
- 第一批实体聚焦：三线制传感器、PLC 输入模块、公共端、24V 电源、电磁阀、气缸、气源压力。

## 优先级路线

### P0：学生端行动闭环

已完成：

- 学习驾驶舱。
- 问答后刷新个人图谱和驾驶舱。
- 基于驾驶舱打开图谱、培养方案、场景、自测。

下一步：

- 讲题后自动生成 1 道复问题。
- 完成任务后让学生上传或填写交付物摘要，并转成图谱证据。

### P1：岗位差距可视化

目标：

- 把岗位能力图谱和个人能力图谱并排对比。
- 明确“岗位要求权重高，但学生证据少”的节点。

轻量实现：

- 新增 `/api/graph/gap?session_id=...`。
- 前端显示 Top 5 岗位差距，不引入重型图谱库。

### P1：知识与资源推荐精细化

目标：

- 同一个问题下，优先推荐最相关的一条知识、一条视频、一条实训任务和一条复测题。

轻量实现：

- 为 `resources.json` 增加 `difficulty`、`duration_minutes`、`use_when`、`ability_ids`、`knowledge_ids`。
- 检索时按能力命中、问题模式、资源类型和时长排序。

### P2：场景库扩展

目标：

- 从 4 个典型现象扩展到 10 个岗位排故场景。

重点场景：

- 电源 24V 正常但传感器灯不亮。
- PLC 输入灯亮但程序地址监控不变。
- 输出点动作但电磁阀不吸合。
- 电磁阀有动作但气缸不伸出。
- 气缸动作慢或卡滞。

### P2：教师确认式岗位图谱更新

目标：

- 让教师导入企业岗位材料后，系统给出能力权重建议。
- 教师确认后进入岗位图谱快照。

轻量实现：

- 保持 JSON 快照，不做爬虫，不做数据库。
- 增强更新建议的理由、证据片段和影响节点。

## 不建议现在做的事

- 不做账号系统、班级后台、权限和数据库。
- 不做完整 LMS。
- 不做自动爬虫持续抓取岗位需求。
- 不做真正多 Agent 自治循环。
- 不训练 DKT/SAKT 等深度知识追踪模型，当前数据量不够。
- 不把知识库扩成完整机电一体化课程体系。

## 评审视角下的讲法

这个产品的创新不在于“有聊天、有题、有图谱”这些单点，而在于把职业新人真实问题转成一个可审计的训练闭环：

```text
真实问题
-> 安全提醒
-> 现场证据追问
-> 直接排故回答
-> 能力/知识缺口定位
-> 学习驾驶舱排序
-> 今日训练与场景练习
-> 自测或复问验证
-> 个人能力图谱更新
-> 教师/师傅带教摘要
```

这个闭环比传统做题系统更贴近岗位训练，也比纯聊天机器人更可控、可解释、可评审。
